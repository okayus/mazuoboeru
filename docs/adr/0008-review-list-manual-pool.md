# 復習は手動の設問プール（Review List）＋通知駆動スペーシング — SM-2 在庫スケジューラを持たない

Phase 3「定着」を、アルゴリズム的 SRS（SM-2）ではなく **Review List**（[[CONTEXT.md]]・UI ラベル "my hot list"）＝**設問単位・手動のプール**＋**[[Drill]]**（1問ずつ解き直し）で実装する。再出題の間隔（スペーシング）はアプリ内に持たず将来の通知（リマインド）に委ね、在庫スケジューラ（`review_state`：ease/interval/due_at）と「今日の復習」due キューは作らない。（2026-06-18 grill で決定。設計のみ確定し、実装は別セッション。）

## Context

Phase 3「定着」は当初、間隔反復（SRS）をアルゴリズム的に実装する想定だった（`docs/data-model.md` の `review_state`＝ease/interval/due_at の SM-2 パラメータ、ロードマップの「今日の復習」キュー・「自己評価で間隔調整」）。実装着手前の grill（2026-06-18）で前提を見直すと、次が分かった。

- 設問は MCQ でサーバー客観採点だが、**復習は私的学習**でランキングも無く、**カンニング／不正な正答率対策は無関係**（[[0002-publish-flow-and-edit-rules]] の客観採点は「公開・対他者」の関心＝公開集計の正直さと採点前の正解秘匿）。よって復習の信号は自己評価でも手動運用でも問題ない。
- 「覚えた／まだ」は専用ウィジェットで取らずとも、ユーザが**「まだ苦手な設問のプール」を手で出し入れする行為**でそのまま表せる（まぐれ正答でも外さなければ残る）。
- 再出題の**間隔（いつ出すか）**は、アプリ内のスケジュール調整画面ではなく**将来の通知（リマインド）機能**に置くのが自然（ユーザ意向。通知設定で interval 的概念を持ちうる）。

## Decision

復習は **Review List**（[[CONTEXT.md]]・UI ラベル "my hot list"）として実装する。

- **設問単位・手動のプール**（`review_list(user_id, question_id)`）。登録はクイズ単位の一括（追加時点の設問をスナップショットで投入）でも設問単位でもよく、**卒業（覚えた）は設問単位で外す**。旧 `favorite`（クイズ単位・#47）は**これに置き換える**。
- **SM-2 在庫スケジューラ（`review_state`：ease/interval/due_at）とアプリ内「今日の復習」due キューは作らない。** プール全体が復習対象。
- 「覚えた／まだ」は**プールへの残留／除外そのもの**で表す（専用評価 UI なし）。客観採点は [[Drill]] 中の即時フィードバックにのみ使い、スケジュールの主信号にはしない。
- [[Drill]]（プールの設問を1問ずつ解き直す）の回答は `review_answer(user_id, question_id, is_correct, answered_at)` に記録し、**ストリーク／活動量に算入**する（[[0006-dashboard-aggregation-semantics]] を本 ADR と同時に改訂）。`attempt`/`attempt_answer` には乗せない（Attempt＝1クイズ通しの意味を保つため）。
- **再出題の間隔（スペーシング）は将来の通知機能に委ねる**。本 ADR 時点ではアプリ内に間隔概念を持たない。

## Considered Options

- **フル SM-2（`review_state`・ease/interval/due_at・4段階自己評価・due キュー）**: 定着測定としては理想だが、専用スケジューラ＋評価 UI＋設問単位の状態遷移を要し、単独ドッグフーディング段階には過大。MCQ は認識ベースでまぐれ正答があり、SM-2 のグレード信号とも相性が中途半端。
- **手動の設問プール（採用）**: ユーザの「苦手設問を手で回す」直感に一致し最小実装。間隔は通知へ分離＝関心の分離。
- **旧 Favorite（クイズ単位）流用**: 粒度が粗く、設問単位の卒業を表せない（却下）。20問中2問だけ苦手でもクイズ単位でしか出し入れできない。
- **ドリルをエフェメラルにし記録しない**: 最小だが「毎日復習してるのにストリークが伸びない」＝学習実態とのズレを生む。設問単位 Review List の目的（苦手設問を毎日回す＝ドリルが主要な学習手段になる）と矛盾するため、記録（`review_answer`）を採用。

## Why

- プロダクトの本義「まず覚える＝反復で定着」の核（反復）は満たしつつ、**最適間隔の最適化は後段（通知）へ分離**できる。
- 客観採点の思想は公開・対他者の関心であり、私的復習には適用されない。手動プール＋自己卒業で十分。
- ドリルを活動量に算入することで、復習が主要な学習手段になってもストリークが学習実態とズレない。

## Consequences

- 旧 `favorite`（クイズ単位・#47・本番稼働中）は **`review_list`（設問単位）に置き換え**る。既存 favorite はドッグフーディングで少量のため「破棄して再登録」で移行（低リスク）。スキーマ変更は `cloudflare-d1-drizzle-migration` 準拠（テーブル追加は安全、`favorite` 撤去は table rebuild）。読み出しは設問の属する quiz を `status='published' AND deleted_at IS NULL` で絞る（非公開化された設問は自然に落ちる＝旧 favorite と同方針）。
- `docs/data-model.md` の `review_state` 行は**当面作らない**。将来アルゴリズム的 SRS が必要になれば別 ADR で `review_state` を再開する（後戻りコストを承知のうえ）。
- [[0006-dashboard-aggregation-semantics]] のストリーク／活動量の源に `review_answer` を加える（同 ADR に追記済み）。正答率は従来どおり活動量フレーミング（オープンブックの再回答も含む・定着の厳密測定はしない）。
- 「間隔反復の“間隔”」は**通知機能の実装まで存在しない**（当面はプール総ざらい）。
- UI ラベル "my hot list"・正準語 "Review List" は**暫定命名で改名可**（[[CONTEXT.md]]）。
