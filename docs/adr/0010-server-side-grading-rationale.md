# サーバー採点の根拠は「正誤の単一真実・即時フィードバック・読みモデル分離」であって競争的 anti-cheat ではない

サーバー側採点（[[0002-publish-flow-and-edit-rules]] の strict 採点）は**維持する**が、その根拠を「カンニング／不正な正答率対策」から **(1) 正誤の単一の真実・(2) 解説の文脈的開示（即時フィードバック）・(3) 読みモデルの分離**へ再定義する。本アプリにはランキング／競争／公開スコアが無く学習は内面的なので、挑戦者に正解が漏れること自体は**セキュリティ上のバグとして扱わない**。（2026-06-18 決定。コード変更なし＝動機と docs の整合のみ。）

## Context

- これまで [[CLAUDE.md]]・[[security.md]]・[[concept.md]]・[[features.md]]・`tech-stack.md` は server 採点を一貫して「カンニング/不正防止」と説明してきた。
- [[0008-review-list-manual-pool]] で既に半分は整理済み: 「復習は私的でランキングも無く**カンニング／不正な正答率対策は無関係**。[[0002-publish-flow-and-edit-rules]] の客観採点は『公開・対他者』の関心＝**公開集計の正直さ**と**採点前の正解秘匿**」。本 ADR はその「公開・対他者」側も見直す。
- **本アプリには正答率ランキング・公開スコア・競争要素を実装する予定がない**（学習は内面的なサポート、クイズ共有は自分を広げるため＝ユーザ意向 2026-06-18）。ダッシュボードの正答率・ストリークは**私的**（[[0006-dashboard-aggregation-semantics]]）で、公開集計（クイズ単位の挑戦数・平均正答率）にも個人を競わせる指標はない。
- つまり「公開集計の正直さ」が守るべき対象（競争的な公開スコア/ランキング）自体が存在しない。残るのは「挑戦者が devtools 等で正解を先に見る」だが、それは**自分の学習を損なうだけ**で他者にも集計にも害が無い＝対策する脅威ではない（自己学習でカンニングは自分を欺くだけ）。

## Decision

サーバー側採点は維持する。ただし根拠を次の3点に再定義し、anti-cheat を動機から外す。

1. **正誤の単一の真実**: 正誤・スコアはサーバが唯一の権威。クライアント計算を信用しないのは「改ざん脅威への防御」ではなく**正しさの一元化（整合性）**のため。
2. **解説の文脈的開示**: 正解・解説は回答後にのみ返す。これは [[CONTEXT.md]] の [[Immediate Feedback]] という**プロダクト機構**であって秘匿ではない。
3. **読みモデルの分離**: 「問題の公開射影（`presenters/quiz.ts` の `publicQuizJson`）」と「採点後の回答レスポンス」は別概念で、正解キー（`is_correct`）・解説は後者に属する。射影が `isCorrect`/`explanation` を含まないのは**設計の綺麗さ**であってセキュリティ境界ではない。

- 「挑戦前に正解が漏れること」は**脅威モデルから除外**する。リグレッションとして緊急対応しない（=「アプリ最大の不変条件」ではない）。
- **コード変更は不要**。現状の `publicQuizJson` 射影・サーバ採点はそのまま「良い設計」として残す（上記 (3) の理由で維持。撤去はしない）。
- 将来ランキング／公開スコア／競争を入れるなら、その時に **anti-cheat を別 ADR で復活**させる（公開集計の正直さが初めて load-bearing になる）。

## Considered Options

- **動機を anti-cheat のまま据え置く**: docs と実態（ランキング無し）がズレ続け、過剰な重要度付け（「最重要不変条件」）が判断を歪める（実際 2026-06-18 のコード点検で射影テストを過大評価する事故が起きた）。却下。
- **サーバ採点をやめてクライアント採点に簡素化**: 正誤の単一真実・即時フィードバックの解説出し分け・射影の綺麗さを失う。却下（採点はサーバのまま）。
- **動機を再定義（採用）**: コード不変・docs 整合・脅威モデルの正直化を同時に満たす。

## Why

- 脅威モデルは「守る価値のあるもの」に対して書く。ランキングが無い以上、正答率の盗み見／改ざんには守る対象が無い。
- サーバ採点の本当の価値（単一真実・即時フィードバック・読みモデル分離）は**ランキングの有無に依存せず常に成立**する。動機をそこへ置く方が頑健。
- 過大な「最重要不変条件」ラベルを外すと、テスト／レビューの優先度が実態に合う（射影テストは必須でなく、**採点配線＝フィードバックの正しさ**のテストが価値）。

## Consequences

- prose docs（[[CLAUDE.md]] / [[security.md]] / [[concept.md]] / [[features.md]] / `tech-stack.md` / `project-status.md`）の「カンニング/不正防止」表現を本 ADR に沿って書き換え、本 ADR を指す。[[CONTEXT.md]] に語「サーバー採点（Server-Side Grading）」を追加し根拠を固定。
- [[security.md]] 脅威モデル表の「不正回答・カンニング」「なりすまし・スコア改ざん」行は、**ランキング不在**を明記し「現状は競争脅威の対象なし／将来ランキング導入時に再掲」と整理。サーバ採点は (1)(2)(3) の理由で残る。
- `presenters/quiz.ts` の射影テストは「セキュリティ不変条件」としては**不要**（コード点検での最優先から除外）。一方、**採点配線**（`already_answered` 冪等・全問回答で finalize・スコア集計）のテストは「フィードバックの正しさ」として価値が残る（[[0008-review-list-manual-pool]] の [[Drill]] が同じ採点経路を再利用するため）。
- `data-model.md` の `attempt_answer.is_correct` 不変（書き込み後に変更しない）は、根拠を「履歴の改変＝不正の防止」から **「履歴の自立性（回答時点の事実を保存し、後のクイズ編集に影響されない）」** へ言い換える。
- 「公開集計の正直さ」は将来ランキング／公開スコアを作る時に初めて必要になる＝その時 anti-cheat を別 ADR で復活させる。
